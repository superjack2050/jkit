# Spec Template

The spec is a **shared contract between human and coding agent**.
Human reviews it for correctness. Agent executes it for precision.
Every section serves both readers.

---

## Template

```markdown
# Spec: [Feature Name]

> Generated by before-build | [Date] | Mode: [codebase|conversation]
> Appetite: [N days/weeks] | Priority: [must-ship / bet]

## 1. Why

[2-3 sentences: why this exists now, what triggered it, what happens if we don't build it.
Human reads this to validate priority. Agent reads this to understand trade-off context.]


## 2. For Whom & Status Quo

**Target user:** [Specific persona — not "developers" but "indie devs using Claude Code
who start 2-3 projects per week and currently write specs by hand or skip them entirely"]

**Current solution:** [How they solve this today — tools, workarounds, manual processes]

**Key pain:** [The #1 friction point in their current approach]

> Human: validate the persona matches your actual target.
> Agent: use this to prioritize FRs — solve the key pain first.


## 3. Overview

[2-3 sentences: what this feature does and how it improves on the status quo.
This is the "after" to section 2's "before".]


## 4. Actors

| Actor | Description | Permission Boundary |
|-------|-------------|---------------------|
| [Role] | [What they do] | [What they can/cannot access] |


## 5. Functional Requirements

Each FR uses a **dual-layer format**: natural language on top (human reads),
implementation spec indented below (agent reads).

**FR-1: [Human-readable description]**
> - File: `src/lib/auth.ts` — create `signJWT(payload): string`, `verifyJWT(token): Payload`
> - File: `src/middleware.ts` — read JWT from httpOnly cookie, reject with 401 if invalid
> - Library: `jose` for JWT operations, token TTL = 7 days
> - Behavior: on success, inject `{ userId, email }` into request context

**FR-2: ...**

### No-go's (Explicitly Out of Scope)

- [Feature X] — deferred to v2
- [Feature Y] — not needed for target persona
- ...

> Why dual-layer: human skims the bold line to confirm intent.
> Agent reads the indented block for exact files, functions, and behaviors.
> No-go's prevent scope creep during implementation.


## 6. Data & State

> What persistent data or state does this feature manage?
> Adapt the format to the project type. Omit if there is no persistent state.

**For web apps / services** — use the project's actual ORM syntax:

// Users — account created on first OAuth login
export const users = sqliteTable('users', {
  id: text('id').primaryKey().$defaultFn(() => nanoid()),
  email: text('email').notNull().unique(),
  name: text('name'),
  createdAt: text('created_at').default(sql`(datetime('now'))`),
});

**For CLI tools** — config file format + persistent state:

// Config: ~/.toolname/config.json
interface Config {
  apiKey: string;
  outputFormat: 'json' | 'csv' | 'table';
  verbose: boolean;
}
// State: ~/.toolname/state.json (persisted between runs)
interface State {
  lastRunAt: string;
  processedIds: string[];
}

**For libraries / SDKs** — core data structures and types:

interface InvoiceData {
  invoiceNumber: string;
  items: LineItem[];
  totalNet: number;
  totalVat: number;
}

**For infrastructure** — configuration schema:

// terraform variables, k8s manifests, or cloud config shapes

> In codebase mode: match existing patterns.
> In conversation mode: pick the format from Tech Constraints, note [ASSUMED] if undecided.
> Human: verify the data design. Agent: use the exact syntax shown.


## 7. Interface Contract

> How does the outside world interact with this feature?
> Adapt the format to the project type. Omit if there is no external interface.

**For web apps / services** — REST / GraphQL / RPC endpoints:

### POST /api/[path] — [one-line description]

**Request:**
{
  "field": "type — description"
}

**Response (2xx):**
{
  "field": "type — description"
}

**Errors:**
| Status | Code | When |
|--------|------|------|
| 400 | INVALID_INPUT | [condition] |
| 401 | UNAUTHORIZED | [condition] |

**For CLI tools** — commands, arguments, and output:

### `toolname generate <input> [options]`

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--output`, `-o` | string | stdout | Output file path |
| `--format` | enum | json | `json`, `csv`, `table` |
| `--verbose`, `-v` | bool | false | Show detailed progress |

**Exit codes:** 0 = success, 1 = input error, 2 = runtime error

**For libraries / SDKs** — public function signatures:

### `generateInvoice(data: InvoiceData, template?: Template): Buffer`

**Parameters:**
| Param | Type | Required | Description |
|-------|------|----------|-------------|
| data | InvoiceData | Yes | Invoice data to render |
| template | Template | No | Custom template, defaults to EU standard |

**Returns:** PDF as Buffer
**Throws:** `ValidationError` if required fields missing

**For infrastructure** — input/output configuration:

Describe the configuration interface (environment variables, config files, CLI flags)
that operators use to control the system.


## 8. File Map

> Human: see the architecture at a glance.
> Agent: know exactly what each file exports, preventing circular deps and scope creep.

| File | Responsibility | Exports |
|------|---------------|---------|
| `src/lib/auth.ts` | JWT sign/verify | `signJWT()`, `verifyJWT()`, `requireAuth()` |
| `src/middleware.ts` | Route guard | Next.js middleware function |


## 9. Acceptance Criteria & Success Metrics

**Acceptance Criteria:**

Each AC has two parts: Given/When/Then (human verifies logic) + verify command (agent self-tests).

| ID | Criteria | Verify |
|----|----------|--------|
| AC-1 | **Given** [precondition] **When** [action] **Then** [result] | `[runnable command]` → `[expected output]` |

> Verify commands are optional in conversation mode (no running environment).
> In codebase mode, every AC should have a verify command.

**Success Metrics:**

| Metric | Target | How to Measure |
|--------|--------|----------------|
| [e.g., spec generation completion rate] | [e.g., > 80%] | [e.g., count projects reaching tasks_ready / total projects] |

> Metrics may require analytics tracking. If so, note in FR section.
> [ASSUMED] if user didn't specify — flag for confirmation.


## 10. Risks

| # | Type | Description | Mitigation |
|---|------|-------------|------------|
| 1 | Edge case | [What could break] | [How to handle] |
| 2 | Rabbit hole | [What looks simple but isn't] | [How to contain scope] |
| 3 | Pre-mortem | [If this fails, most likely because...] | [How to prevent] |

> Three lenses: Edge cases (Stripe), Rabbit holes (Shape Up), Pre-mortem (Lenny).
> Focus on top 3-5 risks. Don't attempt exhaustive coverage.


## 11. Assumptions & Open Investigations

**Assumptions** — reasonable guesses, safe to code against with risk of minor rework:
- [ASSUMED] ...

**Needs Investigation** — no evidence yet, must investigate before coding these parts:
- [NEEDS_INVESTIGATION] ... → blocks Task N, resolved by Task 0
```

## Guidelines

### For Humans
- Read sections **1-3** first (Why, Whom, Overview) — that's the strategic validation.
- Then check **FR bold lines** and **AC table** — that's the tactical validation.
- Review **No-go's** — make sure nothing important was excluded.
- Check **Assumptions** — those are decisions you didn't explicitly make.
- If anything feels wrong, push back before STAGE 4 (task decomposition).

### For Agents
- FR indented blocks are your implementation instructions. Follow them precisely.
- Data & State section is copy-pasteable. Use the exact syntax shown.
- File Map defines your scope. Don't create files outside this map without reason.
- No-go's are your boundaries. Don't implement excluded features.
- AC verify commands are your self-test suite. Run them after implementation.

### Adapting to Project Type
- **Sections 6 and 7 are polymorphic.** Pick the format that matches the project type.
  Each section shows examples for web apps, CLI tools, libraries, and infrastructure.
  Use the one that fits; omit the section entirely if not applicable.
- **Section 4 (Actors)** may have only one actor for CLI tools or libraries. That's fine.
- **Section 8 (File Map)** "Exports" means whatever the file exposes: functions, types,
  commands, config keys, endpoints — not just JavaScript exports.

### General
- **Be specific, not generic.** "System sends email" is bad. "System sends HTML email
  via Resend using the `weekly-report` template to users with `role = admin`" is good.
- **Reference real code.** In codebase mode, say "follows the pattern in
  `src/services/auth.ts:sendVerificationEmail`" instead of "similar to existing patterns".
- **FR IDs are stable.** Tasks reference FR-1, FR-2, etc. Don't renumber after creation.
- **Keep it under 250 lines.** If longer, the feature should be split.
