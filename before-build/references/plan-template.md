# Plan Template

A unified document for a single feature — contains spec (what), design
(how-strategy), tasks (how-atomic), and the ambiguity ledger. Produced by
`/before-build`, consumed by `/build`.

**One file per feature**, replacing the old `spec.md` + `tasks.md` split.
The four parts live together so users can read top-to-bottom and catch
ambiguity in context.

## File Naming

Files are named `plan-<feature-slug>.md` and live at project root.
A project can hold multiple plan files, one per feature.

**Slug rules:**
- Lowercase, alphanumeric + hyphens only
- Spaces / underscores / punctuation → hyphens
- Max 40 characters
- If the feature name is non-English (e.g. Chinese), ask the user for an
  English slug explicitly rather than transliterating

**Slug lifecycle:** determined during PHASE 2/3 once Dim 3 (Goal) is clear,
confirmed with the user before PHASE 5 writes the file. Name is immutable
afterward — a rename would break `/build` references and PHASE 11 matching.

Examples:
- "User Authentication with Google OAuth" → `plan-user-auth-google.md`
- "Invoice PDF Generator" → `plan-invoice-pdf.md`
- "周报系统 for admin" → ask user → `plan-weekly-report-admin.md`

---

## Generation Flow (reference for before-build phases)

1. **PHASE 2/3** — feature slug is derived from Dim 3 Goal, confirmed with user.
2. **PHASE 5 writes PART 1** into `plan-<slug>.md` — the WHAT.
3. **PHASE 6 appends PART 2** — HOW at the strategy level, with structural
   constraints (each D-N must list ≥2 alternatives + blast radius).
4. **PHASE 7 writes PART 4 and pauses for checkpoint** — Ambiguity Ledger
   aggregated (§A/§U/§D) and presented to the user for review. PART 3 is
   deliberately held back until the user confirms.
5. **PHASE 8 locks** the plan (Status: draft → locked).
6. **PHASE 9 appends PART 3** — tasks generated only after the lock,
   guided by PART 1/2 plus the confirmed ledger.
7. **PHASE 11 blast radius** — edits re-cross the appropriate gate on the
   matching plan file and re-sync PART 4 before continuing.

## Guidelines

### For humans
- Skim **§1-3** first — strategic validation (Why / Whom / Overview).
- Then **§5 FR bold lines** + **§9 AC** — tactical validation.
- Then **PART 4 Ambiguity Ledger** — this is where the rework-risk lives.
  Every row is a decision you didn't explicitly make; scrutinize accordingly.
- **PART 3 Tasks** is for agents, but worth skimming to catch scope creep.

### For agents
- PART 1 + PART 2 + PART 4 are your context for every task in PART 3.
- PART 3 task indented blocks are implementation instructions.
- PART 4 is your checklist: any unconfirmed ambiguity in the ledger means
  surface it rather than decide silently.
- Per-task Assumptions field is not optional — if empty, say so explicitly.

### Adapting to project type
- **§6 Data & State** and **§7 Interface Contract** are polymorphic.
  Pick the format matching the project; omit entirely if not applicable.
- **§4 Actors** — single actor is fine for CLI/library.
- **§11 Architecture** — for small features can be one line referencing
  an existing pattern.
- **PART 2 minimum** = §11 + §12. §13 + §14 can be light for simple
  features but should still exist (empty = explicit signal).

### Sizing discipline
- Keep total plan.md under ~500 lines. If larger, the feature should
  likely be split into multiple plans.
- Each task ≤ one Claude session (10-30 min). Split if >5 output files
  or >2 layers.

### Status lifecycle
- `draft` — PART 1/2/4 written, PHASE 7 checkpoint pending
- `locked` — PHASE 7 confirmed, PART 3 being generated
- `in-progress` — `/build` executing tasks
- `complete` — all tasks verified, all AC passed

### Anti-patterns
- **Copy-pasting the template literally.** Adapt — omit polymorphic
  sections that don't apply, don't fabricate content to fill in.
- **Duplicating info between PART 2 §12 and PART 4 §D.** PART 2 is where
  the decision lives with rationale; PART 4 is the index. Keep rationale
  in §12, reference ID in §D.
- **Tasks (PART 3) introducing new library/pattern choices not listed in
  PART 2.** If Task N needs a new dependency not in §12, that's a plan
  gap — surface it as a §D entry and pause.
- **Leaving any of §A / §U / §D as bare empty sections.** Use the
  `_None..._` sentinel so the check is visible.
